1. 배경 및 목적 (Context & Goal)
 * Background: 예비 창업가들은 아이디어 검증을 위해 랜딩 페이지 제작, 도메인 연결, 결제 모듈 연동 등에 불필요한 리소스를 낭비합니다. 이 진입 장벽 때문에 검증 없이 개발에 착수하고, 결국 실패합니다.
 * Product Goal: "10줄의 텍스트(가설)"를 "1개의 완벽한 검증 사이트"로 3분 이내에 변환하여, 사용자가 **'Skin in the Game(실제 행동 데이터)'**을 즉시 확보하게 돕습니다.
 * Key Metric (KPI):
   * Time to Deploy: 가설 입력부터 배포까지 180초 미만.
   * Data Reliability: 봇(Bot) 제외 실제 유효 클릭 데이터 정확도 99% 이상.
2. 타겟 유저 및 페르소나 (User Persona)
 * Primary: 'The Dreamer' (초기 창업가) - 코딩/디자인 능력 없음. 자신의 아이디어가 '대박'이라고 믿음. 빠른 확인 욕구 강함.
 * Secondary: 'The Lean PM' (실무 기획자) - 사내 신규 프로젝트 검증용. 데이터 기반 의사결정 필요.
3. 핵심 사용자 스토리 (User Stories) - MVP Scope
A. 가설 설정 및 생성 (Creation)
 * "사용자는 **XYZ 가설 포맷(적어도 X%의 Y는 Z할 것이다)**에 맞춰 아이디어를 입력할 수 있어야 한다."
 * "사용자는 AI가 생성한 세일즈 카피와 대표 이미지를 미리보기(Preview)에서 확인하고, 마음에 들지 않으면 '재생성' 버튼으로 즉시 수정할 수 있어야 한다." (80/20 법칙: 80% 자동생성, 20% 수정)
B. 가짜 문(Fake Door) 경험 (Validation)
 * "방문자(End User)는 실제 서비스처럼 '구매하기' 또는 '구독하기' 버튼을 클릭할 수 있어야 한다."
 * "버튼 클릭 시, 실제 결제가 진행되는 대신 '서비스 준비 중' 모달이 뜨고, 보상(이메일 등록 시 할인 쿠폰)을 제안받아야 한다."
 * [Critical] "시스템은 이 '클릭' 행위를 단순 방문(Page View)과 구분하여 '전환(Conversion)' 데이터로 별도 저장해야 한다."
C. 결과 분석 (Insights)
 * "사용자는 대시보드에서 노출 수, 클릭 수, 클릭률(CTR), 이메일 확보 수를 실시간으로 확인할 수 있어야 한다."
 * "AI는 수집된 데이터를 바탕으로 'Go(진행)' / 'Drop(폐기)' / 'Pivot(수정)' 중 하나의 명확한 조언을 제공해야 한다."

4. 핵심 기술적 챌린지 및 해결 방안 (Technical Challenges)
Q1. 사용자가 생성한 페이지를 어떻게 즉시 배포(*.flashcheck.site)하는가?
 * Solution: Vercel의 Middleware 기능을 사용합니다.
 * 들어오는 요청의 Host 헤더를 감지하여, 서브도메인이 감지되면 메인 앱의 pages/_sites/[site]/index.tsx와 같은 동적 라우트로 내부 리라이팅(Rewrite)을 수행합니다. 별도의 배포 과정 없이 DB 데이터만 있으면 즉시 페이지가 뜹니다.
Q2. AI가 생성한 디자인이 깨지지 않게 하려면?
 * Solution: **'엄격한 스키마(Strict Schema)'**를 적용합니다.
 * AI에게 HTML 전체를 짜라고 하지 않고, 정해진 JSON 포맷(Title, Subtitle, Feature_1_Title 등)으로 내용만 채우게 합니다. UI 레이아웃은 우리가 미리 코딩해 둔 템플릿 컴포넌트(HeroSection.tsx)에 데이터를 주입하는 방식으로 안전성을 확보합니다.

🎨 1. 유저 흐름 (User Flow)
> 핵심 철학: "Don't Make Me Think." 사용자의 고민 시간을 0으로 수렴시킵니다.
> 
이 흐름은 두 명의 주체, **Creator(기획자)**와 **Visitor(잠재 고객)**의 여정으로 나뉩니다.
Flow A: Creator's Journey (생성 및 배포)
 * 가설 입력 (Input): "적어도 X%의 Y는 Z할 것이다" 템플릿에 빈칸만 채웁니다. (인지 부하 최소화)
 * AI 매직 (Generation): 로딩 스피너 대신 "AI가 시장을 분석 중입니다..."라는 마이크로 카피로 기대감을 조성합니다. (30초~1분)
 * 80/20 에디팅 (Review): 생성된 결과물(카피+이미지)을 확인합니다. 마음에 안 들면 '재생성', 좋으면 '수정' 없이 바로 진행합니다.
 * 즉시 배포 (Deploy): '배포하기' 버튼 클릭 시 project.flashcheck.site 링크가 즉시 생성됩니다.
Flow B: Visitor's Journey (검증 실험)
 * 유입 (Traffic): SNS나 커뮤니티 링크를 통해 랜딩 페이지에 접속합니다.
 * 설득 (Persuasion): AI가 작성한 매혹적인 카피와 이미지를 소비합니다.
 * 결정적 순간 (The Moment of Truth): '구매하기' 또는 '사전 예약' 버튼을 클릭합니다. (이 시점에 데이터 집계 +1)
 * 윤리적 퇴장 (Ethical Exit): 결제창 대신 "준비 중입니다" 모달이 뜹니다.
   * UX Tip: "죄송합니다"가 아니라 "축하합니다! 얼리버드 대상자로 선정되셨습니다"로 프레이밍하여 부정적 경험을 긍정으로 전환하고 이메일을 입력받습니다.

📊 2. 데이터 흐름 (Data Flow)
> 핵심 철학: "Garbage In, Garbage Out." AI가 엉뚱한 소리를 하지 않도록 입력을 통제하고, 결과는 정밀하게 추적합니다.
> 
데이터 흐름은 **생성 파이프라인(Creation Pipeline)**과 **분석 파이프라인(Analytics Pipeline)**으로 구분됩니다.
Data Pipeline 1: 프롬프트 엔지니어링 & 생성 (Creation)
 * Input Refinement: 사용자의 거친 XYZ 입력값을 백엔드에서 다듬습니다. (예: 오타 수정, 문맥 보강)
 * Structured Output: OpenAI API에 response_format: { type: "json_object" }를 강제하여, 디자인이 깨지지 않는 완벽한 JSON 데이터를 받습니다.
Data Pipeline 2: 텔레메트리 & 분석 (Analytics)
 * Client-Side Tracking: 방문자의 브라우저에서 발생하는 이벤트(View, Scroll, Click)를 캡처합니다.
 * Bot Filtering: User-Agent를 분석하여 크롤러나 봇의 클릭은 DB 저장 단계에서 사전에 차단합니다. (데이터 오염 방지)
 * Insight Generation: 수집된 View 대비 Click 비율(CTR)을 계산하여, AI가 "이 아이디어는 될 놈입니다(Go)" 또는 "폐기하세요(Drop)"라고 판단하는 근거로 삼습니다.

💡 3. 데이터 스키마 핵심 전략 (Schema Strategy)
단순한 저장이 아니라 '분석'에 최적화된 스키마를 설계했습니다.
 * projects 테이블의 configuration JSON 컬럼:
   * 페이지의 모든 텍스트, 색상, 이미지 위치 정보를 하나의 JSON 컬럼에 몰아넣습니다.
   * 이유: 스키마 변경 없이 AI가 새로운 디자인 요소를 자유롭게 추가/수정할 수 있도록 유연성을 확보하기 위함입니다.
 * events 테이블의 파티셔닝(Partitioning):
   * 로그 데이터는 기하급수적으로 쌓입니다. 이를 날짜별 혹은 프로젝트별로 파티셔닝하여 조회 속도 저하를 방지합니다.
[UX/Data 전문가의 의견]
단무지님, 이 흐름은 **"최소한의 노력으로 최대한의 진실"**을 얻어내는 데 최적화되어 있습니다.
 * Creator는 복잡한 설정 없이 3번의 클릭(생성-확인-배포)만으로 시장에 진입합니다.
 * Visitor는 실제와 같은 경험을 하지만, 브랜드 신뢰도를 해치지 않는 선에서 부드럽게 퇴장합니다.
 * Data는 봇을 걸러내고 순수한 인간의 욕망(클릭)만을 담습니다.
